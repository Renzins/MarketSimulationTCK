# Vanessa Wind Park — Backtester + Spread Graphs

A static, two-page web tool for the Vanessa / Targale wind park (14 ×
4.2 MW = 58.8 MW, Latvia). Both pages run **entirely client-side** —
the embedded dataset (~2.6 MB) drives all simulation in the browser.
No server, no build step, no install.

The tool is split into two halves, navigable from a top navbar:

| Page              | What it does                                                                                  |
| ----------------- | --------------------------------------------------------------------------------------------- |
| **Backtester**    | Strategy P&L over a configurable simulation window. Levels 1 / 2.                             |
| **Graphs**        | Box plots and heatmaps of the mFRR-vs-DA spread, sliced by Baltic-level wind & solar forecasts. |

## Data layout (`data.js`)

```js
const WIND_DATA = {
  start_iso:   "2025-02-05T00:00:00Z",  // earliest ISP
  n:           43070,                   // total rows after NaN-drop
  step_min:    15,                      // minutes per ISP
  offsets:     [0, 1, 2, 3, ..., 43218],// monotonically increasing 15-min offsets

  // Vanessa-specific (Backtester)
  da_forecast:  [...],   // SciPHER day-ahead P50, MW
  id_forecast:  [...],   // SciPHER intraday P50, MW
  q_pot:        [...],   // actual potential generation, MW (kW÷1000 from CSV)

  // Single-clearing prices (used by Backtester + Graphs)
  p_da:    [...],        // LT day-ahead price (EE/LV/LT share), EUR/MWh
  p_mfrr:  [...],        // mFRR-SA upward (= downward in 99.85% of rows), EUR/MWh
  p_imb:   [..., null, null, ...],  // Latvia final imbalance price, EUR/MWh; null where source missing (mostly April 2026)

  // Baltic-level aggregates (Graphs only)
  baltic_wind_da:  [...], // LV + EE + LT day-ahead wind, MW
  baltic_solar_da: [...], // LV + EE + LT day-ahead solar, MW
  baltic_imb_vol:  [...], // LV + EE + LT imbalance volume, MW (signed: + = surplus)
};
```

**Time range**: 2025-02-05 → 2026-05-01 (≈ 14.5 months, 43 070 ISPs).

**`p_imb`** is `null` for ~6.8 % of rows (mostly April 2026) where the
upstream Latvia imbalance-price file ran out. Engine treats null →
NaN, and L2 simulate() short-circuits the imbalance / flat penalty for
those rows (still includes their DA + mFRR revenue).

## Backtester — strategy & sign conventions

Per ISP (15-min period). Energy = MW × 0.25 h. Positive = money to the park.

```
DA revenue       =   Q_da_sold * P_da                  (≥ 0)
mFRR-up rev      =   Q_up      * P_mfrr                (+ when P_mfrr > 0)
mFRR-dn rev      = − Q_dn      * P_mfrr                (+ when P_mfrr < 0)
Imbalance cost   = − Q_short   * P_imb                 (cost when short)
Flat penalty     = − Q_short   * θ_flat
ISP P&L          = (DA + up + dn − imb − flat) × 0.25

Q_position       =   Q_da_sold + Q_up − Q_dn
Q_short          =   max(0, Q_position − Q_pot)
```

mFRR activation triggers when `|P_mfrr| ≥ 1` EUR/MWh:

- `P_mfrr ≥  1`: mFRR-up activates (offer = `Q_w + max(0, Z·(ID − DA))`)
- `P_mfrr ≤ −1`: mFRR-dn activates (offer = **`Q_da_sold`** — capped at the DA position)
- otherwise: nothing fires.

### Physical constraints (audit-applied)

1. **Whole-MW market quantities.** Balancing market accepts integer-MW
   blocks only, so `Q_da_sold`, `Q_w`, `trusted_rev` and `Q_dn_offer`
   are floored. Fractional MW between `floor(F)` and the actual
   forecast are simply not traded.
2. **mFRR-dn capped at the DA position.** A wind park can drop from
   `Q_da_sold` to 0 but cannot go below 0. Therefore
   `Q_dn_offer = Q_da_sold` (independent of Y), and when `Q_da_sold = 0`
   no mFRR-dn revenue is possible.
3. **Position decreases when mFRR-dn fires.** When mFRR-dn activates,
   promised production becomes `Q_da_sold − Q_dn`, which is why the
   `Q_position` formula has the `− Q_dn` term. mFRR-up and mFRR-dn
   cannot both fire in the same ISP (`P_mfrr` is single-signed).

### Default parameters

| Param             | Default            | Notes                                                      |
| ----------------- | ------------------ | ---------------------------------------------------------- |
| Sim window        | full dataset       | configurable from/to text inputs                           |
| mFRR winsor       | 10 / 90 percentile | clips the −10000 / +10000 EUR/MWh outliers                 |
| Imbalance winsor  | 10 / 90 percentile | same idea, on Latvia imbalance price                       |
| X                 | 30 EUR/MWh         | DA price below which we withhold from DA                   |
| Y                 | 1.0                | fraction of forecast to withhold below X                   |
| Z                 | 1.0                | trust placed on positive ID − DA revisions                 |
| θ_flat            | 30 EUR/MWh         | flat penalty per MWh of shortfall (L2 only)                |

**Setup parameters** (sim window, winsorization) define the experiment
environment — they are NOT touched by the **Reset to naïve** button.
Only the **Market parameters** (X, Y, Z, θ) get reset.

## Graphs — what they show

The Graphs page recreates the spread analysis from the original
ENTSOE deck (with slope-related plots intentionally skipped):

```
spread[i]  =  p_mfrr[i] − p_da[i]                  (EUR/MWh)
regime[i]  = "surplus" if  baltic_imb_vol[i] ≥  surplus_threshold (default +30)
             "deficit" if  baltic_imb_vol[i] ≤  deficit_threshold (default −30)
             "neutral" otherwise (excluded from both regime plots)
```

The seven recreated graphs:

| #     | Title                                                            | Type                  |
| ----- | ---------------------------------------------------------------- | --------------------- |
| 1, 2  | mFRR spread by Baltic Wind Forecast — SURPLUS / DEFICIT          | 1-D box plot          |
| 3, 4  | mFRR spread by Baltic Solar Forecast — SURPLUS / DEFICIT         | 1-D box plot          |
| 5, 6  | Spread by Wind × Solar — DEFICIT / SURPLUS                       | 2-D heatmap (median)  |
| 7     | \|Spread\| by Wind Level — Matched by DA Price Band              | 5-panel grouped boxes |
| 8     | \|Spread\| by Solar Level — Matched by DA Price Band             | 5-panel grouped boxes |
| 9     | \|Spread\| by Renewables (Wind+Solar) Level — Matched by DA Band | 5-panel grouped boxes |

All bin boundaries are **quantile-derived from the current simulation
window**, so changing the bucket count (4 → 8) re-derives the
boundaries and produces evenly-populated bins. Setup controls:

- Simulation date range (DD/MM/YYYY text inputs)
- Winsorize spread (10 / 90 default)
- Surplus / Deficit thresholds (−30 / +30 MW default)
- Bucket counts: Wind, Solar, DA price bands, Levels (matched panels)

## European date format

Date inputs are `<input type="text" placeholder="DD/MM/YYYY">` with an
explicit regex pattern, NOT `<input type="date">`. The browser's
native picker is locale-driven and disagrees with users on the
DD/MM/YYYY vs MM/DD/YYYY rendering — text inputs guarantee the
DD/MM/YYYY display everywhere.

Internally the canonical form is still `YYYY-MM-DD` (ISO).
`parseEU(str)` and `isoToEU(iso)` handle the round-trip and accept
`/`, `.`, `-` separators on input. Plotly's tickformat is set to
`"%d/%m/%Y"` for any time axes.

## Tests

`tests.py` runs 28 audit / regression tests:

```
python tests.py
```

Categories:

- **A — Data integrity** (9 tests): baltic aggregations, spread
  formula, NaN handling, time monotonicity, mFRR up == down.
- **B — Engine invariants** (8 tests): whole-MW rounding, mFRR-dn cap,
  up/dn mutual exclusion, naive value, L1 / L2 known regression
  values, window vs per-ISP consistency, NaN p_imb behavior.
- **C — Spec examples** (2 tests): the original brief's two worked
  Level 2 examples (−322.5 € / +375.0 €).
- **D — Graphs engine** (6 tests): regime classification at the ±30
  threshold, quantile bin balance, SURPLUS spread sign < 0, DEFICIT
  spread sign > 0, Baltic wind / imbalance distribution sanity.
- **E — Schema** (3 tests): all required `data.js` columns present
  and equal-length, `Q_pot` and DA forecast within physical bounds.

The two named regression values (€13,257,221 for L1, €13,367,642 for
L2) are deliberately frozen so any unintentional engine change shows
up immediately.

## Regenerating data.js

`data.js` is built from the source CSV by `preprocess.py`:

```
python preprocess.py
```

The script:

1. Reads only the columns the engines need.
2. Drops rows where ANY of these is NaN: scipher_da_p50_mw,
   scipher_id_p50_mw, lt_dayahead_price_eur_mwh, mfrr_sa_upward_lv,
   wind_park_possible, baltic_wind/solar/imb_vol aggregates. **Does
   not** require `final_imbalance_price_latvia` (so April 2026
   survives).
3. Encodes missing `p_imb` as JSON `null` (engine then converts to NaN).
4. Computes Baltic aggregates (LV + EE + LT) for wind/solar/imbalance.
5. Re-runs the spec's two Level-2 worked examples through the same
   formulas as a sanity check on every regenerate.

## Data refresh workflow

If you replace `main_data_with_imbalance.csv` (e.g. with a longer
file, or one where April finally has imbalance prices), the entire
site adapts automatically:

```
python preprocess.py     # regenerate data.js
python tests.py          # see "frozen regression values" caveat below
```

What re-derives **automatically** from the new dataset:

- Date range header text (DD/MM/YYYY)
- Sim-range and chart-range input bounds (data min/max dates)
- Quantile bin boundaries on every Graphs render
- Winsorization bounds (computed within the sim window)
- All chart axes / tooltips / aggregations
- Naïve baseline (recomputed every parameter change at current θ)
- Backtester decompositions, robustness stats, monthly bars
- ISP counts in regime / dead-band / total

What needs **manual attention** when the data changes substantially:

- **`tests.py` regression values** are frozen on the current dataset
  (L1 = 13,257,221 €, L2 = 13,367,642 €, expected April-row count =
  2,880, expected ~2,911 NaN p_imb entries). After a data refresh
  these should be updated to the new ground truth — `python tests.py`
  will tell you which assertions need new values, then re-freeze them
  to lock in the new dataset's reference numbers.
- **Cache-busting `?v=N`** in `index.html` and `graphs.html`. Bump N
  when deploying so users re-fetch updated `data.js` / JS / CSS.
- **mFRR up vs down assumption**: code uses `mfrr_sa_upward_lv` and
  assumes `mfrr_sa_downward_lv` matches it (99.85% in current data).
  If a refreshed CSV breaks that, re-evaluate the assumption.

If the refreshed CSV's column names change, both `preprocess.py`
`NEEDED` and `tests.py` `usecols` need to be updated — they fail loud
with `KeyError` rather than silently producing wrong numbers.

## Known quirks (for future agents)

These are non-obvious behaviours by design — read before "fixing".

1. **NaN `p_imb` → 0 imbalance cost.** April-2026 rows have no source
   imbalance price (the upstream file ends in March). The engine
   keeps those rows in the simulation (DA + mFRR revenue still
   accrues) but zeros their imbalance + flat-penalty contributions.
   This UNDERCOUNTS L2 cost if you pin the sim window to April only.
   When a future CSV provides April imbalance prices, `p_imb` simply
   stops being NaN and the cost flows through — no engine change
   needed. Behaviour is in `simulate()` / `simulateTotal()` /
   `monthlyAggregation()` (each guards independently — check
   `tests.py` "April ISPs: NaN p_imb → 0 imb cost in L2" if you
   refactor).

2. **Whole-MW market quantities lose fractional MW.** Forecasts come
   in as floats (e.g. 7.2 MW) but the balancing market accepts only
   integer MW. We `floor()` `Q_da_sold`, `Q_w`, `trusted_rev`, and
   `Q_dn_offer`. Up to 1 MW per ISP is therefore "untraded" by the
   strategy — visible mostly with very small forecasts (F < 5 MW).

3. **mFRR-dn capped at `Q_da_sold`, NOT `Q_w`.** A wind park can
   curtail down from its DA position to zero; it cannot go negative.
   When `Q_da_sold = 0` (e.g. Y=1 below X) there is no mFRR-dn
   revenue at all. This was an audit fix — earlier versions of the
   code allowed phantom mFRR-dn revenue with no DA position. See
   `tests.py` "mFRR-dn capped at Q_da_sold".

4. **`Q_position = Q_da_sold + Q_up − Q_dn`.** When mFRR-dn fires,
   the curtailment reduces our promised production. The `−Q_dn` term
   is critical for shortfall accounting to be correct.

5. **mFRR up == down assumption.** `engine.js` uses
   `mfrr_sa_upward_lv` as the canonical mFRR clearing price.
   `mfrr_sa_downward_lv` matches it in 99.85% of rows in the current
   data (i.e. they are the single clearing price). If a future
   dataset breaks this, the engine silently uses upward only.

6. **DA price shared across LV/EE/LT.** We use
   `lt_dayahead_price_eur_mwh` as the price for the Vanessa park
   (Latvia). If LT/EE/LV ever decouple, this needs revisiting.

7. **Browser cache.** Static-server caching is aggressive. The
   `?v=N` query string on `<script>` and `<link>` tags is the manual
   cache buster — bump `N` after a deploy.

8. **`<input type="text">` for dates.** Native `<input type="date">`
   is locale-driven and shows MM/DD/YYYY on US-locale browsers
   regardless of `lang="en-GB"`. We use text inputs with explicit
   DD/MM/YYYY parsing instead. Trade-off: no native calendar picker.

9. **Engine window is global.** `Engine.setWindow(start, end)` is
   module-level state shared across both pages. Both pages call it
   inside their respective `update*()` paths — so the most recent
   page's window "wins" until the other page re-runs. Currently fine
   because rendering is synchronous; not safe for hypothetical
   parallel async sims.

10. **Plotly via CDN.** No SRI hash (Plotly's CDN doesn't publish
    one). If `cdn.plot.ly` becomes unreachable, the site degrades to
    a blank page; charts fail silently. Worth pinning to a vendored
    copy if you care about offline robustness.

## Deploying to GitHub Pages

1. Push this repository to GitHub.
2. **Settings → Pages → Source:** "Deploy from a branch", **Branch:**
   `main`, **Folder:** `/ (root)`.
3. The site will be live at
   `https://<username>.github.io/<repo>/index.html`. The Graphs page
   is at `/graphs.html`.

The site uses cache-busting `?v=N` query strings on every script /
stylesheet reference, so bumping `N` in `index.html` and `graphs.html`
forces clients to re-fetch after a deploy.

## Performance

- All numeric arrays are converted to `Float32Array` at load time.
- Winsorized arrays are cached and re-derived only when bounds /
  window change.
- Backtester sweep grids: L1 sweep (~860 combos × 43 k ISPs) runs in
  ~150 ms. L2 sweep (~9.5 k combos) runs ~3 s, chunked across
  animation frames.
- Time-series chart: bar mode for ranges ≤ 600 ISPs (~6 days),
  scattergl for longer windows. Full-dataset render (43 k points) in
  ~1.7 s.
- Graphs page: full re-compute (7 box-plot / heatmap charts) in
  ~250 ms.

Plotly is loaded from `cdn.plot.ly` (versioned URL). Plotly's CDN does
not publish SRI hashes, so the script tag has none.
