# Vanessa Wind Park — Backtester + Spread Graphs

A static, two-page web tool for the Vanessa / Targale wind park (14 ×
4.2 MW = 58.8 MW, Latvia). Both pages run **entirely client-side** —
embedded datasets (~3 MB main + ~1.2 MB aFRR + ~85 MB lazy-loaded
per-4-s spread) drive all simulation in the browser. No server, no
build step, no install.

The tool is split into two halves, navigable from a top navbar:

| Page              | What it does                                                                                  |
| ----------------- | --------------------------------------------------------------------------------------------- |
| **Backtester**    | Strategy P&L over a configurable simulation window. **Levels 1 / 2 / 3.** Trades against DA + mFRR + aFRR (configurable per-direction split). L3 adds an intra-day oversell + hedge-mFRR-dn speculative leg (S3). |
| **Graphs**        | Box plots and heatmaps of mFRR / aFRR spread (mFRR + aFRR sub-tabs), plus a **mFRR vs aFRR** sub-tab for joint analysis (per-4-s slot-level direction matrix, sign-agreement bars, spread scatter, stats scoreboard). |

## Backtester — strategy & sign conventions

Per ISP (15-min period). Energy = MW × 0.25 h. Positive = money to the park.

```
DA revenue       =   Q_da_sold * P_da                  (≥ 0)
mFRR-up rev      =   Q_up_mfrr * P_mfrr                (+ when P_mfrr > 0)
mFRR-dn rev      = − Q_dn_mfrr * P_mfrr                (+ when P_mfrr < 0)
aFRR-up rev      =   Q_up_afrr * avg_p_pos             (+ when avg_p_pos > 0)
aFRR-dn rev      = − Q_dn_afrr * avg_p_neg             (+ when avg_p_neg < 0)
Imbalance cost   = − Q_short   * P_imb                 (cost when short)
Flat penalty     = − Q_short   * θ_flat
ISP P&L          = (DA + mFRR-up + mFRR-dn + aFRR-up + aFRR-dn − imb − flat) × 0.25
```

### Market activation — gates per market and direction

```
mFRR (per-ISP, binary):
  P_mfrr ≥  1   → mFRR-up activates  (offer = Q_w + max(0, Z·(ID − DA)))
  P_mfrr ≤ −1  → mFRR-dn activates  (offer = Q_da_sold)
  otherwise   → mFRR offer earns 0

aFRR (per-ISP, profitability gate; partial dispatch averaged across the ISP):
  avg_p_pos > 0 → aFRR-up active  (revenue uses Q_up_afrr)
  avg_p_neg < 0 → aFRR-dn active  (system pays park; −Q × negative = positive)
  otherwise   → that direction earns 0 (the wind park wouldn't have bid)
```

The `avg_p_pos > 0 / avg_p_neg < 0` gates are at ISP granularity. They
hold by construction once `data-afrr-15min.js` is built with the
favourable-only filter (see "aFRR market"): `avg_p_pos ≥ 0` and
`avg_p_neg ≤ 0` dataset-wide.

### mFRR ↔ aFRR split — per-direction parameters s_up, s_dn

For each direction we route part of the offered MW to mFRR and the
remainder to aFRR. Whole-MW market constraint preserved via
round-and-remainder so no MW is lost:

```
Q_up_mfrr = round(s_up * Q_up_offer)     Q_up_afrr = Q_up_offer − Q_up_mfrr
Q_dn_mfrr = round(s_dn * Q_dn_offer)     Q_dn_afrr = Q_dn_offer − Q_dn_mfrr
```

`s_up = s_dn = 1` (default) routes everything to mFRR — the engine
collapses to its pre-feature behaviour and the frozen regression
values hold exactly. `s_up = s_dn = 0` routes everything to aFRR. The
two parameters are independent because the per-direction price
economics aren't symmetric (mFRR-up clears only on upside spikes;
aFRR-up earns continuously on positive avg; same asymmetry for
downward).

### Position accounting

```
Q_position  = Q_da_sold
            + (mFRR-up active ? Q_up_mfrr : 0)         + Q_up_afrr × n_pos_fav/225
            − (mFRR-dn active ? Q_dn_mfrr : 0)         − Q_dn_afrr × n_neg_fav/225

Q_short     = max(0, Q_position − Q_pot)
```

mFRR contributes its full Q when activated (binary across the ISP).
aFRR contributes a time-fraction `n_*_fav / 225` — the count of 4-s
slots where the price was favourable for that direction — same factor
that makes the revenue formula correct.

### Physical constraints (audit-applied)

1. **Whole-MW market quantities.** Balancing market accepts integer-MW
   blocks only, so `Q_da_sold`, `Q_w`, `trusted_rev` and `Q_dn_offer`
   are floored. The `s_up` / `s_dn` split also produces whole-MW
   `Q_*_mfrr` and `Q_*_afrr` (round + remainder).
2. **mFRR-dn capped at the DA position.** A wind park can drop from
   `Q_da_sold` to 0 but cannot go below 0. Therefore
   `Q_dn_offer = Q_da_sold` (independent of Y), and when `Q_da_sold = 0`
   no mFRR-dn or aFRR-dn revenue is possible.
3. **Position decreases when downward fires.** When mFRR-dn activates,
   promised production drops by `Q_dn_mfrr`; when aFRR-dn fires
   profitably it drops by `Q_dn_afrr × n_neg_fav/225`. Both terms
   appear with a minus sign in `Q_position`.
4. **mFRR up vs dn mutually exclusive per ISP** (P_mfrr is
   single-signed). mFRR-up vs aFRR-up vs aFRR-dn vs mFRR-dn can co-fire
   freely if their respective gates pass — different MW pools.

## Level 3 — speculative intra-day oversell (S3)

L3 sits on top of L2 (same DA / mFRR / aFRR / imbalance math) and adds
**a per-ISP speculative leg**: when our rolling forecast of the next
ISP's mFRR clearing price is sufficiently below `VWAP1H`, we oversell
up to `X_cap` extra MW on intra-day AND submit a **hedge mFRR-dn bid**
at `VWAP1H + M`. The hedge bid is a stop-loss mFRR-dn offer the wind
park wouldn't normally place — pitched above the typical clearing — so
when it clears it costs us, but it caps the worst-case loss on the
oversold volume at `bid_price · X` per MWh (vs being exposed to
potentially much higher imbalance settlement).

The rolling stats are taken over **`p_mfrr[H − K − L : H − L]`** (raw,
unwinsorized). The lag `L` (default 4 ISPs ≈ 1 hour) models the
real-world publication latency: at 08:55 a trader bidding for the
09:30 ISP can only see settlement prices through ~08:30. Setting `L = 0`
reproduces the legacy no-lag behaviour. `L` is a manually-set
parameter, **not optimised**. Earlier builds used `p_imb` rather than
`p_mfrr` — `p_imb` is the price the wind park *would have paid if
short*, not the actual signal a real trader uses to time an intra-day
oversell; switching to `p_mfrr` aligns the gate with what's observable.

### Algorithm (per ISP H)

```
INPUTS:  VWAP1H[H], p_mfrr[H-K-L : H-L]  (raw, NaN-aware)
PARAMS:  K, L, DA_skip, S_min, sigma_max, X_cap, M
GATES:
  G0: da_sold[H] < DA_skip                            (else skip S3 entirely)
  G1: spread  = VWAP1H[H] − mean(p_mfrr[H-K-L : H-L]) >= S_min
  G2: sigma   = std(p_mfrr[H-K-L : H-L])              <= sigma_max
  G3: X_prop  = floor(X_cap · min(1, (spread - S_min) / S_min)) >= 1
  G4: VWAP1H[H], rolling stats not NaN (need H ≥ K + L)
IF ALL GATES PASS:
  bid_price        = VWAP1H[H] + M
  intraday_rev     = X_prop · VWAP1H[H]                    (always)
  hedge_clears     = p_mfrr[H] <= bid_price                (NB sign!)
  IF hedge_clears:
    curtail_rev    = X_prop · (−p_mfrr[H])                 (signed)
    Q_position    += 0   (curtailment offsets oversell)
  ELSE:
    Q_position    += X_prop                                (shortfall grows)
```

### Sign convention for the hedge bid

`p_mfrr` is **unified across directions** (Latvia single clearing).
A hedge mFRR-dn bid at `bid_price` clears whenever the marginal price
is no more than our ceiling — i.e. `p_mfrr ≤ bid_price`. This is the
stop-loss framing:

- `p_mfrr ≪ 0` (very negative): grid pays us `|p_mfrr|` per MWh
  curtailed — **windfall**.
- `0 < p_mfrr ≤ bid_price`: we pay the grid `p_mfrr` per MWh curtailed
  — **bounded loss** (capped at `bid_price · X` per MWh, cheaper than
  imbalance would have been).
- `p_mfrr > bid_price`: bid doesn't clear; we're exposed to the
  imbalance settlement.

### S3 parameters (defaults)

| Param        | Default | Range slider     | Notes                                                                      |
| ------------ | ------- | ---------------- | -------------------------------------------------------------------------- |
| `K`          | 4       | 2 – 48 ISPs      | Rolling lookback window for the mFRR-price forecast (1 hour default).      |
| `L` (lag)    | 4       | 0 – 24 ISPs      | Publication lag — skip the L most-recent ISPs (unobservable at bid time). **Not optimised.** |
| `DA_skip`    | 50      | 0 – 59 MW        | S3 is skipped on ISPs where `da_sold ≥ DA_skip`. Prevents oversell when the park is already near max DA commitment. **Setting to 0 disables S3 entirely; setting to ≥ 59 turns the gate off. Not optimised.** |

### Dual optimisers

L3 has **two ⚡ Optimise buttons**, by design (mFRR / aFRR market levers
and S3 speculation levers are tuned independently):

- **Optimise market** — sweeps X / Y / Z / s_up / s_dn while holding
  S3 params fixed at their current values.
- **Optimise oversell** — sweeps K / S_min / σ_max / M while holding
  market params, lag L, **X_cap**, and **DA_skip** fixed. Grid runs in
  ~3 s. X_cap is intentionally excluded because the backtest has no
  price-impact term, so any sweep would always select the grid maximum
  (see "Known quirks" below); X_cap must be set by hand from real-world
  liquidity considerations. DA_skip is also a physical / risk gate, not
  an optimisation lever.

- L2 default (`X=30, Y=1, Z=1, θ=30, s_up=s_dn=1`)     = **13,367,642 €**
- L3 with **`s3_X_cap=0`** (S3 disabled)                = 13,367,642 € (= L2 ✓)

### ⚠ Backtest caveat — S3 scales infinitely because price impact isn't modelled

A real, documented finding: with `X_cap` left in the sweep, the
optimum would have landed on the grid maximum every time — selling more
is always net-positive on this dataset. We responded by **pinning
`X_cap` out of the sweep entirely** (see "Dual optimisers" above);
`S_min` and `σ_max` still sometimes settle on the grid edge. The
interior dimensions (`K ≈ 4`, `M ≈ 5`) are genuine optima.

**Why the boundary-optimum is real but unactionable**:

1. **The strategy formula correctly identifies the right opportunities.**
   At every triggered ISP, the model picks "high expected ID-vs-imbalance
   spread in a stable regime". The S_min / σ_max filters are doing their
   job; the optimum simply discovers that, on this dataset, *more*
   opportunities than the defaults gate is profitable.
2. **The backtest has no price-impact term.** Selling 30 MW into the
   Latvian intra-day market would push `VWAP1H` against you, eroding
   the spread the strategy depends on. The model assumes infinite
   liquidity at the quoted VWAP.
3. **No execution fees / spreads**, no counterparty / TSO ratification
   risk, no capacity constraints beyond `X_cap` itself.

Operational reading: the formula is sound for **identifying when** to
oversell; **how much** to oversell needs a separate (out-of-scope)
price-impact model. Recommended manual values for in-the-real-world
use: `S_min = 15–25`, `σ_max = 75–150`, `X_cap = 5–10` MW.

## aFRR market — averaged 15-min prices, favourable-only filter, gates

aFRR runs at 4-second cadence in real life. Pricing every 4-s slot
inside the per-ISP simulation would be too slow for the Backtester's
parameter sweeps, so we **pre-average** AST_POS / AST_NEG to one
number per 15-min ISP via `preprocess-afrr-15min.py`. Two important
twists:

### Favourable-only filter (preprocess-afrr-15min.py)

A wind park bidding sensibly only accepts activation when the price
favours its direction. Inside a single 15-min ISP, the 4-s prices can
swing sign — e.g. AST_NEG = `{−10 €/MWh, +50 €/MWh, NaN×223}`. A
naïve average gives `+0.18 €/MWh`, the engine's gate then drops the
entire ISP, and the wind park loses the `1 MW × 4 s × 10 €/MWh =
0.011 €/MW` it would actually have earned during the favourable slot.

The fix is to filter before averaging:

```
AST_POS values ≤ 0   →  replaced by 0  (we wouldn't bid into money-losing upward)
AST_NEG values ≥ 0   →  replaced by 0  (we wouldn't pay to curtail)
```

After the filter, by construction:
- `avg_p_pos[i] ≥ 0` for every i
- `avg_p_neg[i] ≤ 0` for every i
- the per-direction profitability gate (`avg_p_pos > 0` / `avg_p_neg < 0`)
  is equivalent to "at least one favourable 4-s slot existed in this ISP"

The same filter drives two **favourable counts**:
- `n_pos_fav[i]` — count of 4-s slots in ISP i where AST_POS > 0
- `n_neg_fav[i]` — count of 4-s slots where AST_NEG < 0

These scale the L2 position contribution: dispatched MW =
`Q_*_afrr × n_*_fav / 225`. Distinct from `data-afrr.js`'s `n_pos`
/ `n_neg`, which count ALL non-NaN slots regardless of sign.

### Revenue formula derivation

A 4-s slot's energy share of an ISP is `4/3600` hours, and an ISP is
`0.25` hours. So if a single 4-s slot fires upward at 900 €/MWh and
the rest are NaN:

```
revenue per offered MW
  = 1 MW × 900 €/MWh × (4 s / 3600 s)
  = 1 €
```

With our averaged price:

```
avg_p_pos = max(0, 900) / 225 = 4 €/MWh
revenue per offered MW = 1 MW × 4 €/MWh × 0.25 h = 1 €  ✓
```

The same `1/225` factor appears in the position contribution
(`Q_afrr × n_*_fav / 225`), so revenue and position stay consistent.

### Why the optimiser finds asymmetric optima on this dataset

For this Vanessa data (after the filter):
- median `avg_p_pos` ≈ +16.7 €/MWh — aFRR-up earns continuously, often
  more than mFRR-up's spike-driven earnings
- median `avg_p_neg` = 0 (most ISPs no favourable downward slots);
  when there ARE, the magnitudes are large

The L1 optimiser at default X / Y typically picks `s_up ≈ 0` (route
upward to aFRR) and `s_dn ≈ 1` (keep downward in mFRR), recovering
~150 k € over the symmetric-mFRR baseline. Your dataset may differ —
the per-direction split exists precisely so the optimiser can find
the right mix.

**Four chart families on this sub-tab**:

1. **Summary statistics** (7 tiles) — total slot count, mFRR fire rates,

   Pearson correlation between (winsorized) mFRR spread and aFRR spread.
2. **Joint direction matrix** — 3 × 4 heatmap. Rows: mFRR direction
   inferred from `p_mfrr` sign (Up ≥ +1 / Dead band / Dn ≤ −1).
   Columns differ by mode: slot-mode uses "POS only / NEG only / Both /
   Neither" 4-s slot types (using `AFRR_DATA.n_pos`, `n_neg`, `n_any`,
   `n_total`); ISP-mode uses aFRR-Up-only / Dn-only / Both / Neither
   ISP classification.
3. **Sign agreement bars** — for each direction (POS / NEG), 4-way
   tally of `sign(mFRR spread) × sign(aFRR spread)`. Both-positive +
   both-negative = agreement; mixed signs = disagreement. **Caveat**:
   aFRR-NEG spread is structurally ≤ 0 (since `avg_p_neg ≤ 0` and
   `p_da` typically > 0), so the right-hand chart is heavily dominated
   by "both negative" — informative for confirming that bias rather
   than for new signal.
4. **Spread correlation scatter** — `mFRR spread` (x) × `aFRR spread`
   (y) cloud, with a `y = x` reference line. Diagonal cloud = perfect
   agreement; flat = uncorrelated; anti-diagonal = inverse. Subsampled
   to ≤ 8 k points by stride.

**Winsorisation**: two independent percentile pairs (mFRR + aFRR, both
default 10 / 90). Both are applied to every sign-agreement count,
correlation, and scatter point. Without this, ±10 000 €/MWh outliers
in `p_mfrr` dominate the Pearson term and squash the scatter cloud to
a single pixel. Cap previews show the live clip values per direction
in the Setup card.

**Real findings** from the current dataset (defaults, slot-level):

- **Co-fire UP is strong, co-fire DN is weak.** ~64% of POS-direction
  4-s slots fall inside an mFRR-up ISP; only ~46% of NEG-direction
  slots fall inside an mFRR-dn ISP. The two markets agree more strongly
  on upward needs than downward.
- **Sign-agreement at slot level ≈ 49% for POS direction** — barely
  above what independence would predict (~46%). Slot-level disagreement
  is real, not a definitional artefact (audited via the directly-
  computed sign counts).
- **Pearson correlation ≈ 0.07 raw, ≈ 0.18 with 25/75 winsor.** The
  outlier-dominated raw number underrepresents the central correlation;
  the markets do move together, just weakly.
- **Interpretation**: mFRR-SA (15-min scheduled clearing on forecast
  errors) and aFRR (4-s automatic frequency restoration) respond to
  different signals at different timescales. They agree directionally
  most of the time but their pricing detail differs slot-by-slot.

### Day-type filter (all sub-tabs)

Each sub-tab has an independent **day type filter** (All / Weekends +
holidays / Workdays only). Public-holiday detection runs through the
`date-holidays` plugin for **Latvia, Estonia and Lithuania** — a date
counts as a holiday if any of the three considers it a public holiday.

Mechanics: `Engine.init()` builds a `dayTypeMask: Uint8Array`
(0 = workday, 1 = weekend, 2 = holiday) once at load time, classified
by UTC calendar date. Each engine layer (`GraphsEngine`, `AfrrEngine`,
`AfrrSpreadEngine`, `MfrrAfrrEngine`) reads the mask + the user's
filter state to skip non-matching ISPs in regime classification,
bin-edge derivation, and winsorisation percentiles. If the
date-holidays CDN fails the mask degrades to weekend-only (every
Mon–Fri counts as workday) and the engines log a console.info.

## European date format

Date inputs are `<input type="text" placeholder="DD/MM/YYYY">` with an
explicit regex pattern, NOT `<input type="date">`. The browser's
native picker is locale-driven and disagrees with users on the
DD/MM/YYYY vs MM/DD/YYYY rendering — text inputs guarantee the
DD/MM/YYYY display everywhere.

Internally the canonical form is still `YYYY-MM-DD` (ISO).
`parseEU(str)` and `isoToEU(iso)` handle the round-trip and accept
`/`, `.`, `-` separators on input. Plotly's tickformat is set to
`"%d/%m/%Y"` for any time axes.

## Known quirks (for future agents)

These are non-obvious behaviours by design — read before "fixing".

1. **NaN `p_imb` → 0 imbalance cost.** April-2026 rows have no source
   imbalance price (the upstream file ends in March). The engine
   keeps those rows in the simulation (DA + mFRR revenue still
   accrues) but zeros their imbalance + flat-penalty contributions.
   This UNDERCOUNTS L2 cost if you pin the sim window to April only.
   When a future CSV provides April imbalance prices, `p_imb` simply
   stops being NaN and the cost flows through — no engine change
   needed. Behaviour is in `simulate()` / `simulateTotal()` /
   `monthlyAggregation()` (each guards independently — check
   `tests.py` "April ISPs: NaN p_imb → 0 imb cost in L2" if you
   refactor).

2. **Whole-MW market quantities lose fractional MW.** Forecasts come
   in as floats (e.g. 7.2 MW) but the balancing market accepts only
   integer MW. We `floor()` `Q_da_sold`, `Q_w`, `trusted_rev`, and
   `Q_dn_offer`. Up to 1 MW per ISP is therefore "untraded" by the
   strategy — visible mostly with very small forecasts (F < 5 MW).

3. **mFRR-dn capped at `Q_da_sold`, NOT `Q_w`.** A wind park can
   curtail down from its DA position to zero; it cannot go negative.
   When `Q_da_sold = 0` (e.g. Y=1 below X) there is no mFRR-dn
   revenue at all. This was an audit fix — earlier versions of the
   code allowed phantom mFRR-dn revenue with no DA position. See
   `tests.py` "mFRR-dn capped at Q_da_sold".

4. **`Q_position = Q_da_sold + Q_up − Q_dn`.** When mFRR-dn fires,
   the curtailment reduces our promised production. The `−Q_dn` term
   is critical for shortfall accounting to be correct.

5. **mFRR up == down assumption.** `engine.js` uses
   `mfrr_sa_upward_lv` as the canonical mFRR clearing price.
   `mfrr_sa_downward_lv` matches it in 99.85% of rows in the current
   data (i.e. they are the single clearing price). If a future
   dataset breaks this, the engine silently uses upward only.

6. **DA price shared across LV/EE/LT.** We use
   `lt_dayahead_price_eur_mwh` as the price for the Vanessa park
   (Latvia). If LT/EE/LV ever decouple, this needs revisiting.

7. **Browser cache.** Static-server caching is aggressive. The
   `?v=N` query string on `<script>` and `<link>` tags is the manual
   cache buster — bump `N` after a deploy.

8. **`<input type="text">` for dates.** Native `<input type="date">`
   is locale-driven and shows MM/DD/YYYY on US-locale browsers
   regardless of `lang="en-GB"`. We use text inputs with explicit
   DD/MM/YYYY parsing instead. Trade-off: no native calendar picker.

9. **Engine window is global.** `Engine.setWindow(start, end)` is
   module-level state shared across both pages. Both pages call it
   inside their respective `update*()` paths — so the most recent
   page's window "wins" until the other page re-runs. Currently fine
   because rendering is synchronous; not safe for hypothetical
   parallel async sims.

10. **Plotly via CDN.** No SRI hash (Plotly's CDN doesn't publish
    one). If `cdn.plot.ly` becomes unreachable, the site degrades to
    a blank page; charts fail silently. Worth pinning to a vendored
    copy if you care about offline robustness.

11. **aFRR favourable-only filter is applied AT PREPROCESS TIME, not
    runtime.** Inside a single 15-min ISP, the 4-s prices can swing
    sign — e.g. AST_NEG values `{−10, +50, NaN×223}`. Naïve averaging
    gives `+0.18 €/MWh`, the engine's `avg_p_neg < 0` gate then drops
    the entire ISP, and we lose the favourable-slot earnings. The
    fix is in `preprocess-afrr-15min.py`: AST_POS values ≤ 0 and
    AST_NEG values ≥ 0 are replaced with 0 BEFORE the sum. So
    `avg_p_pos ≥ 0` and `avg_p_neg ≤ 0` dataset-wide BY CONSTRUCTION.
    `n_pos_fav` / `n_neg_fav` are the matching favourable-slot counts
    that scale the L2 position contribution. If you ever change the
    averaging logic, also revisit the synthetic mixed-sign test
    `test_afrr_15min_filter_recovers_mixed_sign_isp` in `tests.py`.

12. **`s_up` and `s_dn` are independent on purpose.** The mFRR-vs-aFRR
    economics differ between directions (mFRR-up clears only on upside
    spikes, aFRR-up earns continuously on positive avg; mFRR-dn fires
    when the system needs to curtail at negative prices, aFRR-dn earns
    when AST_NEG goes negative). On the current dataset the optimiser
    settles at `s_up ≈ 0` (route upward to aFRR) and `s_dn ≈ 1` (keep
    downward in mFRR). A single shared `s` parameter would lose
    ~150 k €/yr of "easy" asymmetric optimum — DO NOT collapse them.

13. **aFRR profitability gate at ISP granularity.** mFRR has a binary
    per-ISP gate (`|P_mfrr| ≥ 1`); aFRR has a similar gate per direction
    (`avg_p_pos > 0` for upward, `avg_p_neg < 0` for downward). When
    the gate fails, the offered volume earns 0 AND contributes 0 to
    the L2 position — the wind park simply didn't bid that direction
    this ISP. This is the "smart bidding" model; without it, an
    unfavourable ISP would generate negative aFRR revenue (the wind
    park paying for the privilege of being curtailed) and the
    optimiser would just learn to stay in mFRR universally.

14. **Day-type filter holidays via CDN plugin.** `date-holidays` v3.28
    is loaded from jsdelivr only on `graphs.html`. The UMD bundle
    exposes `Holidays.default` (NOT `Holidays` directly — newer
    ES-module-flavoured wrapping). `engine.js` probes both shapes.
    If the CDN fails entirely, `Engine.init` logs a console.info and
    falls back to weekend-only classification (every Mon–Fri counts
    as workday). Holiday dates are matched by UTC calendar date — a
    couple of hours of LV/EE/LT timezone slack are absorbed by the
    15-min ISP resolution.

15. **aFRR bar-chart resize on tab switch.** The Graphs page
    pre-renders the 3 aFRR activation-bar charts ~200 ms after page
    load (so switching to the aFRR tab feels instant). Plotly draws
    them into a panel that's still `display: none`, ending up with
    0×0 canvases. The fix is in `graphs-app.js`'s tab handler: after
    activating a panel, it calls `Plotly.Plots.resize` on every
    `.chart` in it. Idempotent and safe on un-rendered charts.

16. **Time-series chart shows 4 bar stacks per direction.** mFRR-up
    (dark green), aFRR-up dispatched (light green), mFRR-dn (dark
    red), aFRR-dn dispatched (light red), with `barmode: "relative"`
    so the same-sign bars stack. The aFRR bar height is the
    DISPATCHED MW (`Q × n_*_fav / 225`), not the offered MW — so a
    30 %-active ISP shows a 30 %-tall bar. The tooltip likewise shows
    offered + dispatched + dispatch-fraction-of-ISP per direction.
    L3 adds two more bars per direction with `marker.pattern.shape =
    "/"` diagonal hatching: **S3 ID oversell** (green hatched, positive
    y) and **S3 hedge curtail** (red hatched, negative y, only when the
    hedge mFRR-dn bid clears). The tooltip's `ISP P&L` line is rendered
    as a colour-coded equation breakdown (e.g.
    `<span style="color:green">150</span> + 30 − 24 = +156 €`) so the
    user can read which components contributed.

17. **S3 hedge mFRR-dn bid is a stop-loss, NOT a windfall trigger.**
    The hedge is an mFRR-dn offer the wind park wouldn't normally place
    — pitched above the typical clearing — so it costs the park when it
    clears, but bounds the loss on the oversold MW. The activation
    condition is `p_mfrr ≤ bid_price` (where `bid_price = VWAP1H + M`,
    typically a positive number). This means the bid clears whenever
    the mFRR-dn marginal price isn't above our ceiling — which includes
    both the windfall case (`p_mfrr < 0`, grid pays us) AND the
    bounded-loss case (`0 < p_mfrr ≤ bid_price`, we pay up to
    `bid_price` per MWh). The earlier draft of S3 had the activation
    condition flipped (`p_mfrr ≤ −bid_price`) which only fired in the
    windfall sub-case; that bug is fixed and the verification trail
    lives in the slot-level comparison's co-fire numbers (hedge clears
    ~89% of S3 oversells at defaults, vs ~6% under the wrong sign).

17b. **S3 imbalance settlement is decomposed but mathematically equivalent
     to one line.** When the hedge bid doesn't clear and the wind park
     ends short, the engine splits the cost between three rows:
     `imb = short_l2 × p_imb`, `flat = short_l2 × θ`,
     `s3_extra_cost = (short − short_l2) × (p_imb + θ)`. By construction
     this sums to `short × (p_imb + θ)` — i.e. exactly the naïve
     "total short at the imbalance settlement price" cost, no double-
     counting, no omission. The decomposition exists so the L3 stats
     panel can attribute cost to L2 vs S3 separately, not because the
     economics differ. Two consequences worth knowing:
     - When `p_imb` is NEGATIVE (system pays the park to be short),
       `s3_extra_cost` goes negative → S3 oversell becomes a windfall,
       because the wind park profits from the shortfall it created.
     - When `p_imb` is NaN (April-2026 rows), all three terms are
       zeroed in lockstep — S3 looks artificially profitable in those
       windows. The L2 NaN-handling caveat (Known quirks #14) extends
       to S3.
     Both behaviours are tested in `tests.py`
     (`test_s3_imbalance_decomposition_equals_naive_short_cost`,
     `test_s3_nan_pimb_zeroes_all_imbalance_terms`).

17c. **`S_min = 0` makes the proportional-sizing formula degenerate.**
     The S3 sizing rule is
     `X_prop = floor(X_cap · min(1, (spread − S_min) / S_min))`.
     When `S_min = 0`, the inner division is `spread / 0 = ±∞`; with a
     positive spread `(spread − S_min) / S_min = +∞ → min(1, ∞) = 1`, so
     `X_prop = X_cap` whenever the spread is positive (gates G2/G4 still
     apply). This is a *feature*, not a bug — it means "no minimum
     spread; just always go full size" — but it makes `S_min` a step
     function around 0. Avoid `S_min = 0` in serious analysis; use
     `S_min ≥ 1` if you want a smooth size response.

17d. **S3 can sell at negative `VWAP1H` if `mean(past p_mfrr)` is more
     negative.** The spread gate uses `spread = VWAP1H − mean(past
     p_mfrr) ≥ S_min`. In ISPs where the recent mFRR settled deeply
     negative (e.g. −100 €/MWh during a series of mFRR-dn clearings),
     even a slightly negative `VWAP1H` (say −5 €/MWh) yields a positive
     spread of +95, so S3 fires. The strategy then SELLS at `VWAP1H` =
     −5 €/MWh — i.e. pays the intra-day market €5/MWh to take the
     power. Whether this is a real opportunity or a data artefact
     depends on the use case; the audit found such ISPs in the dataset
     (e.g. ISP 10349, 2025-05-24 11:15). If a live strategy should
     refuse to sell at negative prices, add a gate `bid_price > 0` or
     `VWAP1H > 0` outside the engine.

18. **S3 optimiser doesn't sweep `X_cap`.** When `Optimise oversell`
    is clicked, the sweep covers (K, S_min, σ_max, M) only and pins
    `X_cap` at the user's current slider value. This is because the
    backtest has **no price-impact term** — selling more is always
    net-positive on this dataset's frozen liquidity assumption, so
    sweeping `X_cap` would just pick the grid max every time. The
    interior dimensions (`K ≈ 4`, `M ≈ 5`) are still genuine optima
    in the live sweep; `S_min` and `σ_max` may still hit boundaries
    occasionally. Live trading must set `X_cap` from a real
    price-impact / liquidity model. The formula correctly identifies
    *when* to oversell; only the *size* needs an outside input.

19. **mFRR vs aFRR comparison is slot-level by default.** When the
    user opens the "mFRR vs aFRR" Graphs sub-tab, `graphs-app.js`
    triggers `loadAfrrPriceData()` if not already loaded; meanwhile
    the ISP-level fallback renders so the page isn't blank. Once the
    4-s file finishes loading, `loadAfrrPriceData`'s onload callback
    calls `scheduleCmpUpdate()` to swap the charts to slot-level
    automatically. In slot mode, `p_mfrr` is broadcast 225× per ISP
    — so n inflates but effective independent observations stay at
    ~n_ISPs. The correlation tile flags this explicitly.

20. **`p_mfrr` direction is inferred from sign at thresholds ±1.**
    The CSV has separate `mfrr_sa_upward_lv` and `mfrr_sa_downward_lv`
    columns but they're equal in 99.85% of rows (single-clearing-price
    market in Latvia). `engine.js` reads `mfrr_sa_upward_lv` as
    `p_mfrr` and classifies direction by sign vs ±1: this is
    equivalent to the wind farm placing standing bids at "I'll
    provide upward for at least +1 €/MWh" and "I'll curtail for at
    least 1 €/MWh paid to me" (= bid threshold −1). Under the unified
    pricing model these are mutually exclusive per ISP (no overlap
    between the +1 and −1 sides). **Note**: "mFRR-up fires" in the
    code means "the wind farm's mFRR-up bid clears" — NOT necessarily
    "the system needed upward globally". The two are nearly the same
    for our park's strategy but not identical.

## Deploying to GitHub Pages

1. Push this repository to GitHub.
2. **Settings → Pages → Source:** "Deploy from a branch", **Branch:**
   `main`, **Folder:** `/ (root)`.
3. The site will be live at
   `https://<username>.github.io/<repo>/index.html`. The Graphs page
   is at `/graphs.html`.

The site uses cache-busting `?v=N` query strings on every script /
stylesheet reference, so bumping `N` in `index.html` and `graphs.html`
forces clients to re-fetch after a deploy.

## Performance

- All numeric arrays are converted to `Float32Array` at load time.
- Winsorized arrays are cached and re-derived only when bounds /
  window / day-type filter change. Cap values are returned even on
  cache hits so the live preview can update without recomputing.
- Backtester sweep grids:
  - L1 sweep over (X × Y × s_up × s_dn) = 41 × 21 × 6 × 6 ≈ 31 k
    combos × 43 k ISPs → ~12-17 s on Opus-tier hardware.
  - L2 sweep over (X × Y × Z × s_up × s_dn) = 41 × 21 × 11 × 4 × 4 ≈
    150 k combos → ~60-90 s, chunked across animation frames.
- Time-series chart: bar mode for ranges ≤ 600 ISPs (~6 days),
  scattergl for longer windows. Full-dataset render (43 k points,
  4 stacked bar series) in ~1.7 s.
- Graphs page: full re-compute (7 box-plot / heatmap charts) in
  ~250 ms; aFRR price plots reuse a cache keyed on (window × bins ×
  winsor × day-type) so toggling direction doesn't re-walk the 8.5 M
  per-4-s entries.
- `Engine.init` builds `dayTypeMask` once at page load (~1 ms for
  43 k ISPs); it's a pure function of timestamps so no recomputation
  is needed when the user toggles the filter.

Plotly is loaded from `cdn.plot.ly` and date-holidays from jsdelivr —
both via versioned URLs. Neither CDN publishes SRI hashes, so the
script tags have none.
